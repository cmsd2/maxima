Proposal: Deferred Breakpoints and Line-Snapping
=================================================

Two changes to Maxima's debugger to improve breakpoint usability,
particularly for IDE/DAP integration.  See breakpoints.txt for
background on the current implementation and its limitations.

Status: IMPLEMENTED.  See git log for the deferred-breakpoints
branch.  Tests in tests/rtest_debugger.mac (23 tests, all passing).


A. Deferred Breakpoints
========================

Goal: allow :break to accept a filename and line number even when
the file has not yet been loaded.  The breakpoint is stored as
"pending" and automatically activated when the relevant function
is defined.

Approach: hook into mdefine (mlisp.lisp), called every time a
function is defined.  The hook must run AFTER mputprop stores the
mexpr property, because break-function needs (mgetl fun '(mexpr
mmacro)) to succeed when resolving.

Changes:

1. New global in mdebug.lisp:

     (defvar *pending-breakpoints* nil)

   Each entry is a list: (file line), where file is a string
   (the filename as provided by the user) and line is a fixnum
   (absolute line number in the file).

2. Modified break-function (mdebug.lisp):

   When given a string filename and the existing symbol scan
   fails to find a match, instead of just printing an error,
   push the request onto *pending-breakpoints*:

     (push (list file li) *pending-breakpoints*)
     (format t "~&Breakpoint at ~a line ~a deferred ~
                (file not yet loaded)~%" file li)

   When the symbol scan succeeds, still set the breakpoint
   immediately as before (no change to the happy path).

3. New function resolve-pending-breakpoints (mdebug.lisp):

   Called after each function definition.  Scans
   *pending-breakpoints* for entries whose filename matches
   the newly-defined function's source file.  For each match,
   calls break-function with the filename and absolute line to
   re-enter the normal resolution path (which now succeeds
   because the function is defined).  Unresolved entries are
   kept in the pending list.

4. Hook location in mdefine (mlisp.lisp):

   resolve-pending-breakpoints is called after mputprop stores
   the mexpr (break-function needs the mexpr to exist):

     (set-lineinfo fnname (cadar fun) body)
     (mputprop fnname (mdefine1 args body) 'mexpr)
     (resolve-pending-breakpoints fnname (cadar fun))

5. New :pending debugger command:

   Shows all pending breakpoints that haven't yet been resolved.

6. Maxima-callable wrappers:

   Five new functions exposed to Maxima for programmatic
   breakpoint management (useful for IDE integration and testing):

     set_breakpoint(fun, li)      - set or defer a breakpoint
     pending_breakpoints()        - return pending list as Maxima list
     breakpoint_count()           - number of active breakpoint entries
     valid_breakpoint_count()     - number of non-stale breakpoints
     clear_breakpoints()          - delete all active and pending breakpoints


B. Line-Snapping for Non-Executable Lines
==========================================

Goal: when a breakpoint is requested on a line with no executable
form, snap to the nearest executable line instead of failing.

Approach: new function nearest-executable-line that wraps
first-form-line with bidirectional search.

Changes:

1. New function nearest-executable-line (mdebug.lisp):

   Tries an exact match via first-form-line first.  If no form
   exists on the target line, searches forward and backward
   alternately, up to 50 lines each way.  Returns two values:
   the form and its actual line number, or nil.

2. Modified break-function (mdebug.lisp):

   Replaced the direct call to first-form-line (and the old
   li=0 retry hack) with nearest-executable-line.  When
   snapping occurs, prints an informational message:

     "Line N has no executable code; adjusted to line M"

3. Stepping behavior — no change:

   The stepping functions (break-step-into, break-step-next)
   already stop on the next form that meval1 evaluates.  The
   "skipping" of non-executable lines is inherent to the
   evaluation model, not a bug.  For IDE/DAP integration, the
   debugger should report the lineinfo of the form where
   execution actually stopped.

D. Not Implemented (future work)
=================================

1. Path normalization:

   The equal filename comparison in break-function and
   resolve-pending-breakpoints is not normalized.  If the user
   passes a different path form than what the loader stores
   (e.g. relative vs absolute, symlinks), matching fails.

   Fix: normalize via truename in add-lineinfo (nparse.lisp)
   at parse time.  This would also fix the existing file-path
   breakpoint bug documented in breakpoints.txt section 5.

2. Macro debugging:

   mdefmacro does not call set-lineinfo or the resolver, so
   deferred breakpoints won't resolve for macros.

3. Breakpoint re-application on redefinition:

   When a function is redefined, existing breakpoints become
   stale — they still exist in *break-points* but reference
   old form objects that will never match in meval1.  This is
   verified by rtest_debugger.mac test 6 (breakpoint_count()
   stays 1 but valid_breakpoint_count() drops to 0).

   The mdefine hook point (where resolve-pending-breakpoints
   is called) is a natural place to add re-application logic:
   before resolving pending breakpoints, check if the function
   already has active breakpoints, save their line offsets,
   delete the stale breakpoints, and re-apply at the same
   offsets on the new body.  Not yet implemented.


E. Alternatives Considered
===========================

1. Hook into set-lineinfo instead of mdefine:

   The initial implementation hooked into set-lineinfo
   (mlisp.lisp:1972).  This failed because set-lineinfo runs
   BEFORE mputprop stores the mexpr property, so break-function
   couldn't find the function definition.  Fixed by moving the
   hook to after mputprop in mdefine.

2. Hook into batchload instead of mdefine:

   Resolve pending breakpoints after an entire file finishes
   loading.  Simpler hook point but less immediate — if file A
   loads file B (via load() inside A), breakpoints in B wouldn't
   resolve until A finishes.  Rejected in favor of per-definition
   resolution.

3. Synthetic marker forms for non-executable lines:

   Inject no-op forms into the parsed tree at lines with no
   executable content, so every line is steppable.  This would
   give true line-by-line stepping but is highly invasive: it
   changes the parser output, inflates expression trees, and
   could interfere with simplification and display.  Rejected
   as too risky for the benefit.

4. Source-file-based stepping (track file:line instead of form
   identity):

   Decouple stepping from form identity entirely and track
   the current (file, line) pair.  Stop whenever it changes.
   This would be a major architectural change to meval1 and
   the break loop.  Could be considered for a future rewrite
   but is out of scope here.

5. Pre-parse files for deferred breakpoints:

   When :break "file" N is called and file is not loaded, parse
   it to extract lineinfo.  But the parsed forms would be
   different objects than the forms produced by the real load,
   so breakpoints couldn't transfer (they match by eq identity).
   Still requires a deferred mechanism.  Adds complexity with
   no real benefit over simply deferring.
